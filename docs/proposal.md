# پیشنهاد فنی و محصولی — اکستنشن تبدیل صدا به متن

> وضعیت: **در حال بحث و تصمیم‌گیری با کاربر — هنوز تأیید نهایی نشده.**
> این سند بر اساس یافته‌های [research-notes.md](./research-notes.md) نوشته شده است.

## ۱. خلاصه‌ی درخواست اولیه

- کلیک روی هر فیلد متنی در هر صفحه‌ی وب → نمایش آیکن میکروفون کوچک.
- کلیک روی آیکن → شروع ضبط، آیکن به حالت "در حال پخش/ضبط" تغییر می‌کند.
- نمایش متن به‌صورت زنده (real-time) در یک پاپ‌آپ متصل به همان آیکن.
- کلیک دوباره → توقف ضبط، آیکن به میکروفون برمی‌گردد، امکان کپی متن.
- **اولویت اول: پشتیبانی دقیق و باکیفیت از زبان فارسی.**

## ۲. معماری فنی پیشنهادی

```
┌─────────────────────────┐      پیام‌رسانی       ┌──────────────────────────┐      پیام‌رسانی      ┌────────────────────────────┐
│   Content Script          │ ───────────────────► │  Background Service      │ ──────────────────► │  Offscreen Document          │
│  (روی هر صفحه‌ی وب اجرا   │                       │  Worker                  │                      │  (chrome-extension://id)     │
│   می‌شود، داخل Shadow DOM) │ ◄─────────────────── │  (هماهنگ‌کننده‌ی state)   │ ◄────────────────── │  getUserMedia +               │
│  - تشخیص فوکوس فیلد        │   نتایج زنده‌ی متن     │  - مدیریت offscreen doc  │  transcript delta    │  SpeechRecognition یا         │
│  - آیکن میکروفون/ضبط       │                       │  - ذخیره تنظیمات         │                      │  MediaRecorder + Cloud STT   │
│  - پاپ‌آپ RTL زنده          │                       │                          │                      │                              │
└─────────────────────────┘                       └──────────────────────────┘                      └────────────────────────────┘
```

**چرا این معماری؟** طبق تحقیق بخش ۱ فایل research-notes، تنها راه دسترسی قابل‌اتکا و بدون prompt تکراری به میکروفون در Manifest V3، عبور صدا از یک Offscreen Document با origin خود اکستنشن است، نه content script. یک صفحه‌ی onboarding یک‌بارمصرف هم برای گرفتن اولین اجازه‌ی میکروفون لازم است.

## ۳. تصمیمات فنی که باید با شما هماهنگ شوند (پایین‌تر به‌صورت سؤال هم مطرح شده)

### الف) استراتژی موتور تشخیص گفتار
با توجه به تأکید شما روی دقت فارسی، این مهم‌ترین تصمیم پروژه است:

| | فقط Web Speech API (رایگان) | ترکیبی (پیشنهاد من) | فقط ابری (OpenAI) |
|---|---|---|---|
| هزینه | صفر | صفر تا خیلی کم (اختیاری) | همیشه هزینه دارد |
| نیاز به API key | خیر | اختیاری | اجباری از روز اول |
| دقت فارسی | نامعلوم، باید تست شود | با ارتقا به cloud engine بهبود می‌یابد | بالاترین سطح موجود (طبق شواهد کیفی) |
| حس «زنده» بودن | عالی (بومی مرورگر) | عالی (پیش‌نمایش رایگان زنده + پاس نهایی دقیق‌تر در صورت فعال بودن cloud) | خوب (با `stream=true` مدل gpt-4o-transcribe) |
| حریم خصوصی | صدا به سرور گوگل می‌رود (نه صرفاً local) | صدا به گوگل و/یا OpenAI می‌رود (بسته به حالت) | صدا به OpenAI می‌رود |

**پیشنهاد من: حالت ترکیبی.** پیش‌فرض روی Web Speech API (رایگان، بدون نیاز به تنظیمات) کار می‌کند تا همه‌چیز out-of-the-box کار کند؛ اما در تنظیمات، کاربر می‌تواند کلید API خودش (مثلاً OpenAI) را وارد کند تا موتور به‌صورت خودکار سراغ `gpt-4o-transcribe` برود — با کیفیت به‌مراتب بالاتر برای فارسی، و هزینه‌ای در حد چند سنت در ساعت.

**مهم:** این جدول شامل چند فرض کیفی است، نه عدد قطعی. طبق دستورالعمل #۳ شما، **پیش از پیاده‌سازی نهایی، در فاز صفر یک تست عملی روی چند جمله‌ی فارسی واقعی بین این دو موتور انجام می‌دهیم** تا با شواهد واقعی (نه حدس) تصمیم بگیریم کدام پیش‌فرض بهتر است.

### ب) نحوه‌ی تحویل متن به کاربر
طرح اولیه‌ی شما: فقط پاپ‌آپ + دکمه‌ی کپی. این طرح امن و ساده است، اما تفاوت مهمی با ابزارهای حرفه‌ای دیکته (مثل Windows Voice Access یا خود Voice In) دارد: آن‌ها متن را **مستقیم و هم‌زمان با صحبت کردن** داخل همان فیلد درج می‌کنند — بدون نیاز به کپی/پیست دستی.

پیشنهاد من: **درج مستقیم به‌عنوان پیش‌فرض** (متن هم‌زمان با صحبت داخل فیلد نوشته می‌شود) + پاپ‌آپ کوچک کنار آیکن که هم زیرنویس زنده نشان می‌دهد و هم دکمه‌های کپی/ویرایش/پاک‌کردن دارد (برای مواقعی که کاربر می‌خواهد متن را جای دیگری استفاده کند، یا فیلد مقصد پیچیده/غیرقابل‌اعتماد است مثل بعضی ادیتورهای rich-text). این را هم به‌صورت سؤال مطرح می‌کنم چون طرح اصلی شما را تغییر می‌دهد.

### ج) محل آیکن میکروفون
متن شما («گوشه‌ی صفحه») می‌تواند دو معنی داشته باشد: گوشه‌ی ثابت مرورگر (یک دکمه‌ی شناور کلی) یا گوشه‌ی همان فیلدی که فوکوس گرفته (مثل Grammarly). پیشنهاد من دومی است چون با جمله‌ی بعدی شما («پاپ‌آپ متصل به همون آیکن») سازگارتر است و وقتی چند فیلد در صفحه هست، ابهامی درباره‌ی «این آیکن مال کدام فیلده» ایجاد نمی‌شود.

## ۴. ویژگی‌های پیشنهادی برای حرفه‌ای و به‌روز شدن (فراتر از طرح اولیه)

این‌ها پیشنهادهایی هستند که خواستید قبل از ساخت مطرح کنم:

1. **پردازش خروجی فارسی**: نرمال‌سازی خودکار با کتابخانه‌ی `persian-tools` — نیم‌فاصله‌ی صحیح، تبدیل حروف عربی به فارسی (ي→ی، ك→ک)، یکسان‌سازی اعداد (فارسی/لاتین، قابل تنظیم).
2. **نشانگر سطح صدا (waveform/level meter)** حین ضبط — اطمینان بصری به کاربر که میکروفون واقعاً صدا را می‌گیرد.
3. **میان‌بر صفحه‌کلید** برای شروع/توقف ضبط بدون نیاز به کلیک ماوس (برای کاربران حرفه‌ای).
4. **صفحه‌ی تنظیمات (Options Page)** با: انتخاب زبان (فارسی/انگلیسی/خودکار)، انتخاب موتور، مدیریت کلید API، حالت درج متن (مستقیم/فقط‌پاپ‌آپ)، تم روشن/تاریک.
5. **فونت فارسی مدرن (Vazirmatn)** به‌صورت bundle‌شده داخل اکستنشن (نه از CDN — چون CSP بعضی سایت‌ها بارگذاری فونت خارجی را مسدود می‌کند و بارگذاری از سرور خارجی برای حریم خصوصی و ریویوی Chrome Web Store هم بهتر نیست).
6. **مدیریت خطا و حالت‌های حاشیه‌ای به‌صورت کامل، نه صرفاً حالت خوش‌بینانه**: نبود اجازه‌ی میکروفون (با راهنمای رفع)، قطعی اینترنت، عدم تشخیص صدا، عدم پشتیبانی سایت (iframe متفاوت‌دامنه)، محدودیت شناخته‌شده روی Google Docs.
7. **پشتیبانی کامل Shadow DOM / iframe هم‌مبدأ / سایت‌های SPA** (به‌جای فرض ساده‌ی یک فرم استاتیک HTML).
8. **دسترسی‌پذیری (Accessibility)**: برچسب‌های ARIA برای آیکن و پاپ‌آپ تا با screen reader هم قابل‌استفاده باشد.
9. (فاز ۲ – پیشنهادی، نه MVP): دستورهای صوتی برای علائم نگارشی («نقطه»، «ویرگول»، «خط جدید») و حالت کاملاً آفلاین با Whisper محلی (WASM) برای کاربران حساس به حریم خصوصی.

## ۵. پشته‌ی فناوری پیشنهادی

- **Manifest V3** + **TypeScript**
- **WXT** (wxt.dev) به‌جای CRXJS/Plasmo — طبق تحقیق، فعال‌ترین و به‌روزترین فریم‌ورک برای اکستنشن در ۲۰۲۶.
- `chrome.offscreen` برای ضبط صدا، `chrome.storage` برای تنظیمات/کلید API (ذخیره‌ی local، نه sync، تا کلید API بین دستگاه‌ها sync نشود).
- کتابخانه‌ی `@persian-tools/persian-tools` برای پس‌پردازش متن فارسی.
- فونت Vazirmatn (bundle محلی).
- رابط کاربری: Vanilla TS + Shadow DOM (سبک، بدون وابستگی سنگین فریم‌ورک UI، تداخل صفر با CSS صفحه‌ی میزبان).

## ۶. فاز صفر — اعتبارسنجی فنی قبل از ساخت کامل (پیشنهادی)

طبق دستورالعمل‌های شما (#۲، #۳، #۷)، پیش از ساخت کامل UI/UX، یک prototype حداقلی می‌سازیم تا این‌ها را با شواهد واقعی (نه حدس) تأیید کنیم:
1. جریان کامل مجوز میکروفون (صفحه‌ی onboarding → Offscreen Document) واقعاً بدون prompt تکراری کار می‌کند.
2. `SpeechRecognition` داخل Offscreen Document با `lang="fa-IR"` واقعاً نتیجه‌ی زنده و قابل‌قبول می‌دهد.
3. مقایسه‌ی کیفی دقت فارسی بین Web Speech API و `gpt-4o-transcribe` روی چند جمله‌ی نمونه‌ی واقعی (رسمی + محاوره‌ای + آمیخته با کلمات انگلیسی/تخصصی).

اگر این فاز مشکلی نشان بدهد، معماری را همان‌جا اصلاح می‌کنیم — نه بعد از ساخت کل UI.

## ۷. مجوزهای اکستنشن (Permissions) که نیاز داریم

- `host_permissions: ["<all_urls>"]` یا معادل — چون قرار است روی *هر* سایتی کار کند (اجتناب‌ناپذیر برای این نوع ابزار؛ باید در توضیحات Chrome Web Store دلیلش شفاف نوشته شود).
- `offscreen`, `storage`. (توجه: پرمیژن جداگانه‌ای به‌نام "microphone" در مانیفست وجود ندارد.)
